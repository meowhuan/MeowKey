# 安全说明

## 1. 当前安全基线

MeowKey 仍然是开发期认证器，因此安全策略分成两档理解：

- `debug` 构建
  面向开发联调，默认不安全。
- `hardened` 构建
  关闭 Debug HID，适合作为后续安全加固的基线，但仍不是量产级设计。
- `probe` 构建
  用于板卡识别与 preset 生成，不应被当作长期使用固件。

## 2. 这次已经收紧的点

### 2.1 Debug HID 可在构建时关闭

之前固件默认总会暴露 Debug HID。现在已经可以通过构建参数关闭：

- CMake: `-DMEOWKEY_ENABLE_DEBUG_HID=OFF`
- PowerShell: `-DisableDebugHid`

这能直接切掉最危险的一条开发后门。

### 2.2 `pinUvAuthToken` 不再持久化，且生命周期更窄

运行时 PIN token 现在只存在于当前上电会话中，不再当作持久化状态写入 Flash；并且每次 `getPinToken` 都会重新签发短生命周期令牌。

这意味着：

- 重启后旧 token 失效
- 旧 token 会按短生命周期失效
- 攻击者无法仅凭离线读 Flash 长期复用旧 token

### 2.3 比较逻辑改成常量时间风格

PIN 相关 HMAC / 哈希比较路径已经改成常量时间风格，避免最直接的比较短路泄漏。

### 2.4 未实现的权限 token 子命令被显式拒绝

`getPinTokenWithPermissions` 之前会走到不完整语义。现在直接返回不支持，避免“看起来支持，实际上没做权限校验”。

### 2.5 存储层已升级到事务化 A/B 提交

主凭据区现在带有 `generation/crc` header，并按 A/B 槽提交；`signCount` journal 也会绑定当前主区代际与校验值。

这减少了：

- 掉电时把整个主区写坏的风险
- journal 与主区压实后错配的风险
- 高频断言带来的整区擦写

## 3. 仍然存在的主要风险

### 高风险

#### 3.1 Debug 构建仍然不适合直接分发

只要 Debug HID 开着，设备就允许开发工具：

- 读取诊断日志
- 列出凭据摘要
- 清空凭据存储

这不是生产边界。

#### 3.2 凭据私钥仍明文存放在设备 Flash

当前私钥、`credRandom`、PIN 哈希都保存在普通 Flash 保留区中。开发期可以接受，但量产前必须重新审视。

#### 3.3 私钥与 PIN 哈希仍明文存放在设备 Flash

当前事务化已经补上了掉电一致性，但并没有解决“静态敏感材料明文躺 Flash”的问题。

### 中风险

#### 3.4 当前的用户在场（UP）仍是基础实现

当前已经为外接 GPIO 按键预留了持久化配置接口；但对 `meowkey_rp2350_usb` 这类把 `BOOTSEL` 接到 Flash `CS` 的板子，不应把 `BOOTSEL` 当运行时 UP。
默认构建会保留协议兼容性，但跳过真实物理确认。

但这仍然不是量产级的人机交互方案。

#### 3.5 没有量产级用户验证（UV）模型

当前 UV 只是一条开发期 PIN 语义，不包含：

- 独立受信输入
- 硬件按键确认
- 安全元件

#### 3.6 UV 仍然只是 PIN，不是更强的本地验证

当前 UV 仍然只是一条开发期 PIN 语义，不包含：

- 独立受信输入
- 硬件按键确认以外的更强本地验证
- 安全元件

### 低风险 / 待完善

#### 3.7 Secure boot 仍未启用

仓库已经有签名镜像入口，但默认没有：

- OTP 公钥哈希
- 回滚底线
- 量产恢复流程

#### 3.8 敏感中间缓冲区仍未全面清零

这次只补了 `clientPIN` 的关键路径。其余加密临时数据仍值得进一步系统化清理。

## 4. 使用建议

### 4.1 开发联调

可以使用 Debug 构建，但应默认认为：

- 主机环境是可信的
- 设备不是保密边界
- 数据可以随时被擦除

### 4.2 对外分发前

至少做到：

1. 使用 `hardened` 构建，关闭 Debug HID。
2. 审查并重构凭据存储层。
3. 明确 UP / UV 硬件策略。
4. 定义 secure boot 和 rollback policy。
5. 做真实设备级恢复与升级测试。

## 5. 不应该误判的事情

以下结论目前都不成立：

- “能注册和断言，所以已经是安全认证器”
- “CI 能过，所以已经适合量产”
- “支持签名镜像入口，所以 secure boot 已经完成”
- “有 PIN，所以已经具备完整 UV”

当前更准确的描述是：MeowKey 已经具备可调试的认证器功能骨架，并开始具备最基本的发行边界，但距离量产安全设计还有明显距离。
